Technician RAG Assistant

Techbot — een zelf-gehoste RAG-chatbot waarmee technici in natuurlijke taal vragen kunnen stellen aan een kennisbank van industriële documentatie (kraan- en hijsmanuals, drive/PLC-programmeergidsen, sensor-datasheets, prijslijsten) en gegronde, geciteerde antwoorden krijgen met inline diagrammen en tabellen uit de bron-PDF's. Volledig gebouwd op een zelf-gehoste Docker-stack (geen SaaS-RAG-platform), end-to-end: van PDF-ingestie en multimodale extractie tot vector-opslag, agentische orchestratie, en een aparte operator-UI.Techbot — a self-hosted RAG chatbot that lets technicians ask natural-language questions against a knowledge base of industrial documentation (crane and hoist manuals, drive/PLC programming guides, sensor datasheets, price lists) and get grounded, cited answers with inline diagrams and tables pulled from the source PDFs. Built as a fully self-hosted Docker stack (no third-party SaaS RAG platform), end to end: PDF ingestion, multimodal extraction, vector storage, agentic orchestration, and a purpose-built operator UI.

Het probleemThe problem

Kritische technische kennis over kranen en hijsen zit verspreid over honderden PDF's, drive-handleidingen en prijslijsten. Off-the-shelf document-loaders verliezen precies dat wat deze documenten bruikbaar maakt: tabellen worden OCR'd met cijferfouten, diagrammen sneuvelen of raken losgekoppeld van hun context, en gescande documenten leveren dubbele, verminkte tekst op. Bij veiligheidskritische inhoud — belastingtabellen, remafstellingen, bedradingsschema's — is een fout antwoord erger dan geen antwoord. Het systeem werd daarom rond één principe gebouwd: nooit stilzwijgend een waarde verzinnen of overschrijven. Elke stap die met onzekere OCR- of visie-output werkt corrigeert alleen met sterker bewijs, laat anders het origineel intact — en de chat-laag is expliciet geïnstrueerd dat een hedge acceptabel is, een fout getal niet.Safety-critical technical knowledge about cranes and hoists lives scattered across hundreds of PDFs, drive manuals, and price lists. Off-the-shelf document loaders lose exactly what makes these documents useful: tables get OCR'd with digit errors, diagrams get dropped or divorced from their surrounding text, scanned documents produce garbled, duplicated text. For safety-critical content — load charts, brake settings, wiring diagrams — a wrong answer is worse than no answer. The system was built around a single governing principle: never silently fabricate or overwrite a value. Every stage that touches uncertain OCR or vision output only corrects with higher-confidence evidence, otherwise leaves the original untouched — and the chat layer is explicitly told a hedge is acceptable, a wrong number is not.

De aanpakThe approach

Twee Docker-planes, bewust gescheidenTwo Docker planes, deliberately separated

Het project draait op twee Docker Compose-stacks verbonden via een gedeeld netwerk — een ingest-plane (Qdrant vector-store, custom FastAPI-ingestion-service, Docling voor image-descriptie, nginx reverse proxy, statische file-server voor extract-images) en een user-plane (Flowise voor agentic retrieval, Open WebUI als chat-frontend, een custom upload-portaal voor operators). Bewust gescheiden zodat de (relatief fragiele en geheugenhongerige) ingestion-pipeline geherstart, gerebuild of gedebugged kan worden zonder de live chatbot plat te leggen.The project runs across two Docker Compose stacks joined by a shared network — a data/ingest plane (Qdrant vector store, a custom FastAPI ingestion service, Docling for image descriptions, an nginx reverse proxy, a static file server for extracted images) and a user-facing plane (Flowise for agentic retrieval, Open WebUI as chat frontend, a custom upload portal for operators). Deliberately separated so the (comparatively fragile, memory-hungry) ingestion pipeline can be restarted, rebuilt, or debugged without taking down the live chatbot.

De ingestion-pipeline (custom, bovenop unstructured)The ingestion pipeline (custom-built, on top of unstructured)

Eén PDF in → een lijst RAG-ready chunks eruit, via zes geordende stages:One PDF in → a list of RAG-ready chunks out, through six ordered stages:

  1. Layout-extractie — hi_res/YOLOX layout-model + pdfminer text-layer, met page-nummer, coördinaten en herkomst (layout-model vs. text-layer) per element.Layout extraction — hi_res/YOLOX layout model + pdfminer text layer, carrying page number, coordinates, and provenance (layout-model vs. text-layer) per element.
  2. Noise-filtering, in strikte dependency-volgorde — drop OCR'd logo-garbage misgeclassificeerd als titels; verzoen dubbele text-layers op gescande PDF's (elke layout-elementtekst wordt herbouwd uit best-matching pdfminer-lijnen, waardoor OCR-fouten "gratis" gefikst worden zonder een tweede OCR-pass); collabspeer overige near-duplicates per pagina; strip inhoudsopgaves, pagina-nummers en herhaalde headers/footers (>40% van pagina's).Noise filtering, in strict dependency order — drop OCR'd logo/URL garbage misclassified as titles; reconcile duplicate text layers on scanned PDFs (each layout element's text is rebuilt from its best-matching pdfminer lines, fixing OCR errors "for free" without ever running a second OCR pass); collapse remaining near-duplicates per page; strip tables of contents, page numbers, and headers/footers that appear on >40% of pages.
  3. Tabel-visie-verrijking — pagina's met een tabel krijgen die regio uitgeknipt en naar een vision-model gestuurd, specifiek om OCR-cijferfouten te fixen (met name Europese decimale komma's die OCR omzet in duizendtal-scheidingen, bv. "4,5" → "45"). De vision-respons wordt gevalideerd (weigeringen en niet-goed-gevormde <table>-output worden afgewezen) voordat ze de originele OCR-tekst mag vervangen — een slechte of lege vision-respons overschrijft nooit een echte waarde.Table vision enrichment — pages containing a table get that region cropped and sent to a vision model specifically to fix OCR digit errors (notably European decimal commas that OCR turns into thousands separators, e.g. "4,5" → "45"). The vision response is validated (rejects refusals, anything that isn't a well-formed <table>) before it's allowed to replace the original OCR text — a bad or empty vision response never overwrites a real value.
  4. Figuur-detectie — een aparte DocLayNet-pass detecteert en knipt elke diagram/foto-regio per pagina, onafhankelijk van de minder betrouwbare image-extractie van het layout-model zelf.Figure detection — a separate DocLayNet pass detects and crops every diagram/photo region per page, kept independent of the text-layout model's own, less reliable image extraction.
  5. Image-verrijking — elke figuur krijgt snelle Tesseract-OCR plus één vision-model-call voor een retrieval-geoptimaliseerde caption. Bewust een goedkoop model hier, want captions zijn beschrijvend, niet numeriek (in tegenstelling tot tabel-transcriptie, waar correctheid echt telt).Image enrichment — every figure gets fast Tesseract OCR plus one vision-model call for a retrieval-optimized caption, deliberately using a cheap model here since captions are descriptive, not numeric (unlike table transcription, where correctness has real stakes).
  6. Content-addressed image-opslag — figuren worden bewaard op een pad afgeleid van document/pagina/figuur-index, waardoor re-ingestie van een geüpdatete PDF idempotent is (zelfde URL's, files gewoon overschreven) in plaats van orphan-images te verzamelen.Content-addressed image storage — figures are persisted at a path derived from document/page/figure-index, making re-ingestion of an updated PDF idempotent (same URLs, files just overwritten in place) instead of accumulating orphaned images.

Chunking-strategie: één chunk per PDF-pagina. Tekst wordt per pagina samengevoegd (een lopende sectietitel wordt doorgedragen naar pagina's zonder eigen heading), en gedetecteerde figuren op die pagina worden in dezelfde chunk gegooid als [Figure: caption] + OCR-tekst + image-URL. Zo krijgt een antwoord dat een diagram nodig heeft dat diagram altijd samen met de verklarende tekst — in plaats van als losse vector-store-entry die de LLM geen reden heeft om te noemen.Chunking strategy: one chunk per PDF page. Text is joined per page (a running section title carries forward onto pages with no heading of their own), and any figures detected on that page are appended into the same chunk as [Figure: caption] + OCR text + image URL — so an answer that needs a diagram always retrieves it alongside the explaining text, rather than as a disconnected vector-store entry the LLM has no reason to reference.

Agentische RAG-orchestratie (Flowise AgentFlow)Agentic RAG orchestration (Flowise AgentFlow)

De chat-flow is geen enkele retrieve-then-generate-call — het is een kleine agent-graph met query-rewriting, een relevance-gate, en bounded self-correction:The chat flow isn't a single retrieve-then-generate call — it's a small agent graph with query rewriting, a relevance gate, and bounded self-correction:

Optimize Query → Retrieve (Qdrant, top-K) → Check Relevance (LLM judge)
                                                 ├ Relevant   → Rerank → Generate Answer
                                                 └ Irrelevant → Regenerate Question
                                                                    → retry (max 2x)
                                                                       → Explain Not Found
  • Bounded retry-loop met een aparte give-up-pad. Oorspronkelijk retourneerde een uitgeputte retry-budget gewoon een hardcoded string. Ik verving dat door echte state-tracking (een retryCount flow-variabele + een deterministische condition-node) die routeert naar een aparte LLM-node die een natuurlijke, on-brand uitleg genereert van waarom niks gevonden is. Tijdens dat werk ontdekte ik een platform-eigenaardigheid: Flowise's AgentFlow-engine vereist dat alle statische inkomende edges van een node vuren voor die node draait. Het give-up-pad terug bedraden naar de bestaande "Generate Answer"-node deed de flow stilletjes stallen. Bevestigd via execution-traces, opgelost met een dedicated terminal-node.Bounded retry loop with a dedicated give-up path. Originally, exhausting the retry budget just returned a hardcoded string. I replaced that with real state tracking (a retryCount flow variable + a deterministic condition node) that routes to a dedicated LLM node producing a natural, on-brand explanation of why nothing was found — while discovering a real platform constraint: Flowise's AgentFlow engine requires all of a node's static incoming edges to fire before it executes. Wiring the give-up path back into the existing "Generate Answer" node silently stalled the flow. Confirmed via execution traces, fixed with a dedicated terminal node.
  • Prompt-scoped correctheidsregels. De generation-prompt codeert harde regels die uit echte failure-modes zijn geleerd: nooit een "zie tabel N"-referentie parafraseren alsof het de inhoud zelf is, een relevant diagram altijd inline tonen in plaats van beschrijven, en "niet gevonden" onderscheiden van "gevonden maar alleen een toepasbaar/aanverwant model" met een expliciete hedge in plaats van valse zekerheid of over-hedging.Prompt-scoped correctness rules. The generation prompt encodes hard rules earned from real failure modes: never paraphrase a "see table N" reference as if it were the content itself, always show a relevant diagram inline rather than describing it, and distinguish "not found" from "found but only an applicable/adjacent model" with an explicit hedge rather than either false confidence or over-hedging.
  • Performance-tuning, gevalideerd (en deels teruggedraaid) met échte metingen. Ik profileerde de relevance-check-stap en zag dat die de volledige retrieved chunks — inclusief grote embedded image-caption/URL-metadata die de relevance-judgment nooit echt nodig had — bij elke query opnieuw meestuurde, op ~30–58K tokens per call. Ik bouwde een slimmed-down projectie voor die ene stap en valideerde met een scripted A/B-benchmark (20 representatieve vragen tegen beide flow-versies via de prediction-API, vergelijking van latency, token count en answer-equivalentie). De benchmark ving een echte regressie op (een te agressieve early version die chunks te ver truncateerde en zo genuine false "not found"-antwoorden veroorzaakte), die werd geroot-caused en gefikst. Toen de totale end-to-end latency-winst onder meting niet stand hield (de token-besparing vertaalde zich niet betrouwbaar in snellere responses), rolde ik de change terug — kwaliteit boven een onbewezen speed-gain, backed by data i.p.v. aanname.Performance tuning, validated (and partly reverted) with real measurement. I profiled the relevance-check step and found it was re-sending the entire retrieved chunk set — including large embedded image-caption/URL metadata the relevance judgment never actually needed — on every single query, at ~30–58K tokens per call. Built a slimmed-down projection for that one step, validated with a scripted A/B benchmark (20 representative questions run against both flow versions via the prediction API, comparing latency, token count, and answer equivalence). The benchmark caught a real regression (an early, too-aggressive version that truncated chunk text and caused genuine false "not found" answers on in-scope questions), which was root-caused and fixed. When the total end-to-end latency win didn't hold up under measurement (the token savings didn't reliably translate to a faster overall response), the change was rolled back to the original, proven-reliable configuration — prioritising answer quality over an unproven speed gain, backed by data rather than assumption.
  • Veilig production-flow-editen. Elke wijziging aan de live agent-flow wordt eerst gebouwd en geverifieerd tegen een disposable geconeerde kopie van de flow via de prediction-API — happy path én edge cases — voordat ze naar de productie-flow gepromoveerd wordt, met de pre-change versie bewaard als named backup-rij voor instant rollback.Safe production-flow editing. Every change to the live agent flow is built and verified against a disposable cloned copy of the flow via the prediction API — happy path and edge cases — before being promoted to production, with the pre-change version preserved as a named backup row for instant rollback.

Upload-portaal (operator-tooling)Upload portal (operator tooling)

Een kleine FastAPI-service + SQLite-backed job-queue + vanilla-JS frontend, gebouwd na een incident waarbij een naïeve retry-loop tegen een permanent-broken conditie (uitgeput API-quotum) dagenlang onnodige spend verbrandde voordat iemand het merkte. Nu:A small FastAPI service + SQLite-backed job queue + vanilla-JS frontend, purpose-built after an incident where a naive retry loop against a permanently-broken condition (exhausted API quota) burned through days of unnecessary spend before anyone noticed. Now:

  • Classificeert elke ingestion-failure als retry (transient — 5xx, network), skip (alleen deze file — 4xx) of fatal (quota-exhaustion, corrupted document store). Een fatal error pauzeert de hele queue in plaats van door te knallen op de rest van de files met dezelfde onherstelbare fout, duidelijk zichtbaar in de UI met een one-click resume zodra de onderliggende oorzaak gefikst is.Classifies every ingestion failure as retry (transient — 5xx, network), skip (this file only — 4xx), or fatal (quota exhaustion, corrupted document store). A fatal error pauses the entire queue rather than continuing to burn through remaining files with the same unrecoverable error, surfaced clearly in the UI with a one-click resume once the underlying cause is fixed.
  • Herstelt cleanly van een mid-upload service-restart (elke job die bij boot nog "processing" was, is mid-flight gekild en wordt automatisch her-queued, nooit stil vastzittend).Recovers cleanly from a mid-upload service restart (any job still "processing" at boot was killed mid-flight and gets automatically re-queued, never left silently stuck).
  • Verwijdering is een two-phase cleanup: verwijder de Flowise document-store-loader en sweep Qdrant onafhankelijk op matching vectors per filename, want sommige documenten dateren van vóór het portaal en zitten alleen direct in de vector-store zonder loader-record.Deletion is a two-phase cleanup: remove the Flowise document-store loader and independently sweep Qdrant for any matching vectors by filename, since some documents predate the portal and only exist directly in the vector store with no corresponding loader record.

Operationele constraints waar ik omheen gebouwd hebOperational constraints engineered around

15 GB host-memory budget. De ingest-container draait één Gunicorn-worker met een --max-requests-cap — native allocations van torch/onnx/tesseract/pdfium tijdens hi-res OCR/layout/vision worden niet volledig vrijgegeven door Python's GC, dus de worker wordt periodiek gerecycled om RSS-groei te bounden in plaats van jacht te maken op de onderliggende native-library-leak. Container-mem_limit's werden bewust verstrengd van waardes groter dan totale host-RAM (een no-op safety-limit die één lekkende container de hele host liet neerhalen) naar een gemeten plafond, geïnformeerd door een echte OOM-kill op een 888-pagina PDF.15 GB host-memory budget. The ingest container runs a single Gunicorn worker with a --max-requests cap — native allocations from torch/onnx/tesseract/pdfium during hi-res OCR/layout/vision processing aren't fully released by Python's GC, so the worker is periodically recycled to bound RSS growth rather than chasing the underlying native-library leak. Container mem_limits were deliberately tightened from values larger than total host RAM (a no-op safety limit that let one leaking container take down the whole host) down to a measured ceiling, informed by an actual OOM-kill on an 888-page PDF.

De ingestie-pipeline is idempotent en gescheiden van de user-facing chatbot, wat betekent dat documenten toegevoegd, geüpdatet of verwijderd kunnen worden zonder downtime. Technicians ask a plain-language question — including specific model numbers like "STAHL SH66" — and get back an answer with the correct steps, the source reference, and the relevant diagram from the original manual. Wat ik geleerd hebWhat I learned

Nooit stil overschrijven bij veiligheidskritische data. De hardste designbeslissing was ook de simpelste te uiten: een onzekere vision-respons mag nooit een echte waarde vervangen. Dat principe stuurde de architectuur — elke stage die met OCR/vision werkt heeft een validatie- of best-evidence-check, en de chat-prompt vraagt liever een hedge dan een fout getal. Voor load-tabellen en remafstellingen is dat het verschil tussen "goed genoeg voor productie" en "juridische aansprakelijkheid".Never silently overwrite in safety-critical data. The hardest design decision was also the simplest to phrase: an uncertain vision response must never replace a real value. That principle drove the architecture — every stage touching OCR/vision has a validation or best-evidence check, and the chat prompt asks for a hedge over a wrong number. For load charts and brake settings, that's the difference between "good enough for production" and "legal liability".

Meten, niet aannemen — en durven terugdraaien. De token-optimalisatie op de relevance-check leek een duidelijke winst tot ik ze echt tegen benchmarks aanhield. De token-besparing vertaalde zich niet in latency-winst, en een early version veroorzaakte zelfs false "not found"-antwoorden. Uiteindelijk rolde ik het terug — een teleurstellende conclusie, maar de juiste. "Ik denk dat dit sneller is" is geen argument tegenover "ik heb 20 vragen A/B-getest".Measure, don't assume — and be willing to revert. The relevance-check token optimisation looked like a clear win until I actually benchmarked it. The token savings didn't translate to latency, and an early version was even causing false "not found" answers. I ended up rolling it back — a disappointing conclusion, but the right one. "I think this is faster" doesn't beat "I A/B-tested 20 questions".

Fatal vs. retryable classificatie is geen theoretische hygiëne. Een naïeve retry-loop tegen een uitgeput API-quotum kostte échte euros voordat het gemerkt werd. Sindsdien: elke ingestion-failure wordt geclassificeerd als retry, skip of fatal, en fatal pauzeert de hele queue. Die verandering zou zichzelf terugverdiend hebben na één incident.Fatal vs. retryable classification isn't theoretical hygiene. A naive retry loop against an exhausted API quota cost real money before anyone noticed. Since then: every ingestion failure is classified as retry, skip or fatal, and fatal pauses the whole queue. That change would have paid for itself after one incident.

Platform-gedrag confirmeren via traces, niet via documentatie. Flowise's AgentFlow-engine bleek te vereisen dat alle statische inkomende edges van een node vuren voordat die node draait. Dat staat nergens duidelijk beschreven; ik kwam erachter omdat mijn flow silently stalde en execution-traces het lieten zien. Sindsdien: bij elke abstractielaag die je niet zelf gebouwd hebt, is een trace-driven confirmation goedkoper dan een aanname.Confirm platform behaviour via traces, not docs. Flowise's AgentFlow engine turned out to require that all of a node's static incoming edges fire before that node runs. That isn't documented clearly anywhere; I found it because my flow silently stalled and execution traces revealed it. Since then: for any abstraction layer you didn't build yourself, a trace-driven confirmation is cheaper than an assumption.

Live systemen editen zonder ze te breken. Elke change op de live agent-flow gaat eerst door een cloned copy via de prediction-API — happy path én edge cases — voor die in productie mag. De pre-change versie blijft als named backup-rij staan voor instant rollback. Dat proces voelt langzaam, maar het is sneller dan één keer een productie-incident debuggen.Editing live systems without breaking them. Every change to the live agent flow goes through a cloned copy via the prediction API — happy path and edge cases — before it's promoted to production. The pre-change version stays as a named backup row for instant rollback. The process feels slow, but it's faster than debugging one production incident.
